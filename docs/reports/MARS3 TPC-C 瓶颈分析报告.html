MARS3 TPC-C 瓶颈分析报告

基于 172.16.100.143:/root/automan 的 BenchmarkSQL 结果、runBenchmark 阶段资源采集、perf report，以及 MARS3/heap update 路径源码分析。

测试对象YMatrix MARS3 master-only
数据库主机192.168.100.30
数据规模100 warehouses / 15 min
报告日期2026-06-30

总体判断

MARS3 在本轮 TPC-C 中的瓶颈不是磁盘写满，而是高频 UPDATE 被拆成多次 delta btree、rowstore、MARS3 btree index 和 update-chain 回填后，CPU、run queue、buffer pin 和 LWLock 被系统性放大。

100 并发时 CPU 已经接近饱和；500 并发后吞吐下降、延迟暴涨、postgres 进程 wait 升高；1000 并发时有效吞吐进一步塌陷，延迟 p95 达到 82 秒量级，并出现阶段性 CPU 空转与等待交替。

最关键的证据

  • tpmC 从 100 并发的 7317.81 降到 500 的 3158.26，再降到 1000 的 1660.07。
  • 500 并发整体延迟 p95 为 15.22 秒，1000 并发整体延迟 p95 为 82.46 秒。
  • iowait 基本为 0%，主盘 sdb 平均 util 只有 9.51% 到 24.22%。
  • perf 热点稳定出现 _BTGetRoot、ReadBufferExtended、ReadBuffer_common、LWLockAcquire、PinBuffer。

4. MARS3 与 heap update 的结构性差异

MARS3 update 路径

  1. TupleUpdateExtended() 先调用 TupleDeleteInternal(..., update=true) 删除旧版本。
  2. 旧 tuple 状态写入 DeltaItem，DeltaItem 存在 DeltaTree 中。
  3. DeltaTree 是内部 btree，查旧版本需要 root-to-leaf 搜索。
  4. 新版本通过 rowstore 追加写入，并立即获得 ctid。
  5. 为新 tuple 插入 MARS3 btree index，随机 TPC-C update 很难只走 rightmost fastpath。
  6. 最后 SetNewRowCtid() 再查一次旧 DeltaItem，回填 next ctid。

heap update 路径

  1. heap 也写新版本，但 update chain 和可见性信息在 heap tuple header 中。
  2. 如果没有修改索引相关列且同页有空间，可以 HOT update。
  3. HOT update 后旧 tuple 设置 hot-updated，新 tuple 设置 heap-only。
  4. table AM handler 会避免为 heap-only tuple 插入新 index entries。
  5. 未改 key attrs 时，heap 可以使用较弱 tuple lock。

4.1 为什么 MARS3 更烧 CPU

差异点 heap MARS3 对 TPC-C 的影响
旧版本状态 写在 heap tuple header 的 xmax/cmax/infomask/t_ctid。 写 DeltaItem，DeltaItem 在 DeltaTree btree 中。 每次 update 增加 delta btree 搜索和 buffer lock。
索引维护 符合条件时 HOT update，不插入新的索引项。 固定 delete + insert，新版本需要插入 MARS3 btree index。 余额、数量、计数、ytd 等非主键更新仍会放大索引路径。
update chain 旧 tuple 的 t_ctid 指向新 tuple。 SetNewRowCtid() 回填旧 DeltaItem 的 next。 同一次 update 至少两次访问旧 DeltaItem。
tuple lock 可根据是否修改 key attrs 使用较弱锁。 当前直接 LockTupleExclusive。 高并发下更容易放大 tuple lock、multixact 和 xid 检查成本。
临界区接触面 理想路径主要锁旧 heap page，同页 HOT 最轻。 delta btree、rowstore page、MARS3 btree index、delta next 回填。 同一条 SQL 接触更多 buffer 和 LWLock。

5. duplicate key 日志对结果的影响

三档 run 均由 automan 判定为 success，并且 BenchmarkSQL 产出了正式 tpmC。但日志中确实存在大量 duplicate key value violates unique constraint "bmsql_oorder_pkey"。这类错误会影响 TPC-C New Order 语义，应在正式对外结论中单独标注。

6. 瓶颈链路复盘

  1. TPC-C 高频 UPDATE 触发 MARS3 的 delete + insert + index insert + next ctid 回填路径。
  2. DeltaTree btree 查询 让旧版本可见性和 update chain 不再是 heap tuple header 本地操作，而变成内部 btree 操作。
  3. rowstore 新版本追加写 需要排他锁数据页，并为每个 MARS3 btree index 插入新项。
  4. 高并发放大 root/leaf/delta page 争用，perf 中出现 LWLockAcquire、PinBuffer、ReadBuffer_common 高占比。
  5. CPU 和调度队列先被打满，随后 backend 等待比例上升，事务延迟出现秒级到百秒级长尾。
  6. 吞吐下降后 IO 利用率不升反降，进一步证明瓶颈在内部 CPU/lock 路径，而不是块设备。

7. 建议

优先级 动作 目的
P0 先定位 bmsql_oorder_pkey duplicate key 来源，确认是 BenchmarkSQL 配置、sequence 初始化、并发隔离，还是 MARS3 路径问题。 避免在异常语义下解释 tpmC。
P0 用 50、100、150、200、300 terminals 细化拐点，不要只看 500/1000。 找到 MARS3 从可用区进入退化区的真实阈值。
P1 补一组相同参数的 heap 与 MARS3 对照，并记录 HOT update 命中情况、索引插入次数、WAL 量。 量化 MARS3 update 路径相对 heap 的固定成本。
P1 对 500 并发重新采集完整 perf script 或 flamegraph，并保留未截断脚本。 进一步分解 delta btree、MARS3 btree index、rowstore 写入、xid/multixact 的占比。
P2 评估 MARS3 update 是否能引入 heap 类似的 key attrs 判断、弱 lockmode、HOT-like 或 delta item 缓存/合并策略。 减少每次 UPDATE 固定走多次 btree/buffer 临界区的结构性成本。
