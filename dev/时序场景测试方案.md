# 时序场景测试方案

## 范围

本方案描述 Automan 新增 `ts` 测试类型的第一阶段设计。第一阶段只覆盖三类场景：

```text
ts-write
ts-query
point-query
```

这三个 stage 和现有 TPC-C 测试是同级测试项。每个 stage 都可以单独执行，都会生成独立 run 产物，并出现在 `automan list` 中。

第一阶段保留 `compress_threshold` 矩阵，但暂不计算复杂压缩率。压缩效果先用表数据大小体现。用户在 `list` 中按 `compress_threshold` 对比同一列 `Table Data Size`，即可看到不同阈值下的存储大小差异。

## 用户流程

用户先生成时序测试配置：

```bash
./configure -t ts -c ym-mars3 -o automan.yml
```

然后编辑 `automan.yml`，确认数据库、Kafka、stage、写入时长、并发、topic 和矩阵参数。

用户不再需要手动执行：

```bash
./bin/validate -i automan.yml
```

`check.yml` 会自动先做 validate，再做远端环境检查：

```bash
./check.yml -i automan.yml
```

Kafka topic 检查单独执行：

```bash
./automan kafka-check -i automan.yml
./automan kafka-check -i automan.yml --apply
```

执行所有配置的时序 stage：

```bash
./ts.yml -i automan.yml
```

只执行某个 stage：

```bash
./ts.yml -i automan.yml -s ts-write
./ts.yml -i automan.yml -s ts-query
./ts.yml -i automan.yml -s point-query
```

进度、结果列表和报告沿用现有命令：

```bash
./automan progress
./automan list
./report.yml -i automan.yml
```

## 配置结构

时序测试配置使用：

```yaml
benchmark: ts
```

所有可矩阵化的参数统一使用数组格式。即使只有一个值，也写成数组。

示例：

```yaml
all:
  vars:
    benchmark: ts
    job_id:

    ts_stages: [ts-write, ts-query, point-query]

    compress_threshold: [1200, 4096, 8192, 32768]

    kafka:
      host: 172.16.100.214
      port: 22
      user: root
      password: "bigdata@yMatrix.cn"
      kafka_bin: /opt/kafka_2.13-4.1.2/bin
      bootstrap_server: localhost:9092
      topic: iot_vehicle_raw_test
      partitions: [36]
      replication_factor: 1
      recreate_topic: true

    mxgate:
      user: mxadmin
      password: "mxadmin"
      workdir: /home/mxadmin/automan
      binary: mxgate
      source: kafka
      transform: plain
      writer: stream
      delimiter: "|"
      format: csv
      time_format: unix-ms
      stream_use_gzip: "yes"

    ts_write:
      table: iot_vehicle_raw
      workers: [20]
      rate_per_worker: [10000]
      vins: [10000]
      vin_interval: [10s]
      duration_seconds: [300]
      producer_fast_mode: true
      producer_lead_seconds: 20
      monitor_interval_seconds: 5
      lag_drain_timeout_seconds: 600
      require_zero_lag: true

    ts_query:
      dependency_mode: reuse
      rounds: [3]
      warmup_rounds: [1]
      timeout_seconds: 7200

    point_query:
      dependency_mode: reuse
      rounds: [3]
      warmup_rounds: [1]
      sample_size: [1000]
      timeout_seconds: 7200
```

密码如果需要加引号，统一使用双引号。生成模板不要使用单引号。

## 主机角色

第一阶段尽量减少配置项，主机角色按下面规则固定。

执行机是：

```text
172.16.100.143:/root/automan
```

它负责编排 Automan、生成计划、执行控制逻辑、归档结果。

Kafka 机器由 `kafka.host` 配置：

```text
172.16.100.214
```

它负责 topic 管理、Kafka producer、Kafka offset 和 lag 采集。

mxgate 运行在 YMatrix master 机器上。用户在目标配置里填写的 YMatrix IP 就视为 master IP。第一阶段不单独配置 mxgate IP，避免把流程搞复杂。

mxgate 使用 `mxgate.user` 登录 YMatrix master，通常是 `mxadmin`：

```bash
ssh mxadmin@<ymatrix_master_ip>
cd /home/mxadmin/automan/work/ts/<run_id>
mxgate start --config mxgate.conf
```

## Kafka topic 处理

Kafka topic 不能写死，必须从配置读取：

```yaml
kafka:
  topic: ...
```

`./automan kafka-check -i automan.yml` 只检查，不修改 Kafka。

程序步骤：

```text
1. 读取 automan.yml 中的 Kafka 配置。
2. SSH 到 kafka.host。
3. 检查 kafka.kafka_bin/kafka-topics.sh 是否存在。
4. 检查 kafka.bootstrap_server 是否可连接。
5. 检查 kafka.topic 当前是否存在。
6. 打印 --apply 将要执行的 delete/create/describe 命令。
```

`./automan kafka-check -i automan.yml --apply` 才会真正准备 topic：

```bash
export KAFKA_BIN=/opt/kafka_2.13-4.1.2/bin
export PATH=/usr/local/go/bin:$KAFKA_BIN:$PATH
$KAFKA_BIN/kafka-topics.sh --bootstrap-server "$BOOTSTRAP" --delete --topic "$TOPIC"
sleep 5
$KAFKA_BIN/kafka-topics.sh --bootstrap-server "$BOOTSTRAP" --create --topic "$TOPIC" --partitions "$PARTITIONS" --replication-factor "$REPLICATION_FACTOR"
$KAFKA_BIN/kafka-topics.sh --bootstrap-server "$BOOTSTRAP" --describe --topic "$TOPIC"
```

其中 `$BOOTSTRAP`、`$TOPIC`、`$PARTITIONS`、`$REPLICATION_FACTOR` 都来自 `automan.yml`。

正式执行 `ts-write` 时不自动删除 topic。topic 的删除和重建只由 `kafka-check --apply` 控制。

## 生产和消费时序

写入阶段采用“先生产，20 秒后启动消费”的策略。

默认流程：

```text
1. 准备 YMatrix 表。
2. 生成 mxgate.conf，但暂不启动 mxgate。
3. 启动资源采集。
4. 在 Kafka 机器上启动 producer。
5. 等待 ts_write.producer_lead_seconds，默认 20 秒。
6. 在 YMatrix master 上启动 mxgate。
7. producer 和 mxgate 同时运行。
8. producer 到达 ts_write.duration_seconds 后停止。
9. mxgate 继续运行，直到 Kafka lag 归零，或达到 lag_drain_timeout_seconds。
10. 停止 mxgate。
11. 校验写入数据。
12. 采集表数据大小。
```

这样可以观察短暂 Kafka 积压之后，mxgate 从 Kafka 消费并写入 YMatrix 的追赶能力和稳定写入能力。

`ts_write.duration_seconds` 控制 producer 总运行时长。`producer_lead_seconds` 包含在这个时长内。例如 `duration_seconds=300`、`producer_lead_seconds=20`，表示 producer 总共运行 300 秒，mxgate 在第 20 秒启动。

## ts-write

用户命令：

```bash
./ts.yml -i automan.yml -s ts-write
```

每个矩阵组合执行一轮。第一阶段的核心矩阵是 `compress_threshold`。

程序步骤：

```text
1. 为当前矩阵项生成 run_id。
2. 从目标配置解析 YMatrix master IP。
3. 从 kafka.topic 解析 Kafka topic。
4. 在 YMatrix 中创建或重建目标表。
5. 用当前 compress_threshold 渲染 MARS3 DDL。
6. 渲染当前表和 topic 对应的 mxgate.conf。
7. 启动数据库、Kafka、mxgate 相关采集器。
8. 在 Kafka 机器上启动 producer。
9. 等待 producer_lead_seconds。
10. 在 YMatrix master 上启动 mxgate。
11. 持续采集 producer 指标、Kafka offset、consumer lag、mxgate 状态、mxgate 日志和主机资源。
12. duration_seconds 到达后停止 producer。
13. 等待 lag 归零。
14. 停止 mxgate。
15. 查询 YMatrix 表的行数、最小时间、最大时间、VIN 数量。
16. 查询 YMatrix 表数据大小。
17. 写入 run 结果摘要。
```

表名带上 `compress_threshold`，避免不同矩阵项互相覆盖：

```text
<base_table>_ct1200
<base_table>_ct4096
<base_table>_ct8192
<base_table>_ct32768
```

`ts-write` 结果字段：

```text
stage
compress_threshold
kafka_topic
target_table
duration_seconds
producer_lead_seconds
target_qps
producer_ack_qps
producer_attempted
producer_errors
written_rows
distinct_vins
min_timestamp
max_timestamp
final_lag
max_lag
table_data_size_bytes
status
```

## ts-query

用户命令：

```bash
./ts.yml -i automan.yml -s ts-query
```

默认依赖模式：

```yaml
ts_query:
  dependency_mode: reuse
```

`reuse` 表示复用已有 `ts-write` 写入的数据。程序会检查预期表是否存在、是否有数据。如果缺失，stage 失败，并提示用户先执行 `ts-write`，或把依赖模式改为 `auto-write`。

可选依赖模式：

```yaml
ts_query:
  dependency_mode: auto-write
```

`auto-write` 表示如果某个矩阵项缺少数据，程序可以自动为该矩阵项先跑 `ts-write`，再执行查询。

程序步骤：

```text
1. 从 automan.yml 解析矩阵项。
2. 对每个 compress_threshold 解析预期表名。
3. 检查表是否存在且有数据。
4. 如果 dependency_mode=auto-write 且数据缺失，先执行当前矩阵项的 ts-write。
5. 从表中抽样 VIN 和时间范围。
6. 执行 warmup 查询。
7. 启动查询阶段采集器。
8. 执行配置轮数的时序范围查询。
9. 记录每条查询的耗时、返回行数和错误。
10. 停止采集器。
11. 写入 run 结果摘要。
```

典型时序查询：

```sql
select *
from public.iot_vehicle_raw_ct1200
where vin = :vin
  and "timeStamp" >= :start_time
  and "timeStamp" < :end_time
order by "timeStamp";
```

`ts-query` 结果字段：

```text
stage
compress_threshold
target_table
rounds
warmup_rounds
query_count
avg_ms
p50_ms
p95_ms
p99_ms
rows_returned
errors
table_data_size_bytes
status
```

## point-query

用户命令：

```bash
./ts.yml -i automan.yml -s point-query
```

依赖模式和 `ts-query` 一致。

程序步骤：

```text
1. 从 automan.yml 解析矩阵项。
2. 对每个 compress_threshold 解析预期表名。
3. 检查表是否存在且有数据。
4. 如果 dependency_mode=auto-write 且数据缺失，先执行当前矩阵项的 ts-write。
5. 从表中抽样 vin 和 timestamp。
6. 将抽样参数写入 point-query-input.csv。
7. 执行 warmup 点查。
8. 启动查询阶段采集器。
9. 执行配置轮数的点查。
10. 记录耗时、是否命中、返回行数和错误。
11. 停止采集器。
12. 写入 run 结果摘要。
```

典型点查 SQL：

```sql
select *
from public.iot_vehicle_raw_ct1200
where vin = :vin
  and "timeStamp" >= :timestamp
  and "timeStamp" < :timestamp + interval '10 seconds'
order by "timeStamp"
limit 1;
```

`point-query` 结果字段：

```text

compress_threshold
target_table
sample_size
rounds
warmup_rounds
query_count
avg_ms
p50_ms
p95_ms
p99_ms
hit_rate
errors
table_data_size_bytes
status
```

## ts_stages 全量执行

如果 `automan.yml` 中配置：

```yaml
ts_stages: [ts-write, ts-query, point-query]
```

用户执行：

```bash
./ts.yml -i automan.yml
```

Automan 按配置顺序执行 stage。推荐的矩阵执行顺序是：

```text
ct1200:  ts-write -> ts-query -> point-query
ct4096:  ts-write -> ts-query -> point-query
ct8192:  ts-write -> ts-query -> point-query
ct32768: ts-write -> ts-query -> point-query
```

这样每个 `compress_threshold` 的数据生命周期更集中，不需要先把所有数据都写完，再集中跑查询。

如果用户指定单个 stage：

```bash
./ts.yml -i automan.yml -s ts-query
```

则只执行该 stage，但仍遍历所有配置的矩阵项。后续如果需要只跑某个阈值，可以再增加矩阵过滤参数。

## 异常处理

`kafka.topic` 必填。缺失时 validate 失败。

mxgate 固定在 YMatrix master IP 上启动。如果 `mxgate` 命令不存在，`check.yml` 在执行测试前失败。

producer 启动后，如果 mxgate 在等待 20 秒后启动失败，程序停止 producer，并将当前 run 标记为 failed。

如果 `require_zero_lag=true`，且 Kafka lag 在 `lag_drain_timeout_seconds` 内没有归零，`ts-write` 标记为 failed。

如果查询 stage 使用 `dependency_mode: reuse`，但目标表不存在或为空，stage 标记为 failed，并提示用户先执行 `ts-write` 或改用 `auto-write`。

## 暂缓内容

第一阶段不实现复杂压缩率公式，只记录表数据大小。

第一阶段不实现 TPC-H、TPC-DS 和 AP 分析场景。

第一阶段不引入单独的 mxgate host 配置，mxgate 使用 YMatrix master IP。

第一阶段不在 `ts-write` 中自动删除 Kafka topic。topic 的修改只通过 `automan kafka-check --apply` 完成。
